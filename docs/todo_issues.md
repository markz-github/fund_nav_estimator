# 待处理问题

## 1. 新增基金请求可能因超时误报失败

### 现象

前端新增基金时，页面可能提示“新增基金失败，请检查基金代码或后端日志”，但后端稍后可能已经成功完成基金添加和官方净值同步。

### 当前原因判断

- 前端 Axios 请求超时时间当前为 `15000ms`。
- 后端调用 `akshare` 获取基金名称和官方净值时，当前没有设置显式超时时间。
- 如果 `akshare` 接口响应较慢，可能出现前端先超时，而后端仍在继续处理的情况。

### 影响

- 用户可能以为基金添加失败。
- 实际数据库中可能已经写入基金。
- 再次添加同一基金时，可能看到数据已经存在或列表突然出现该基金。

### 后续处理建议

- 将新增基金请求的前端超时时间适当调大，例如 `30000ms`。
- 后端为 akshare 调用增加明确的超时、重试和错误捕获。
- API 返回更细分的错误信息，例如：
  - 基金不存在
  - 数据源超时
  - 数据源返回空
  - 数据库写入失败
- 前端根据错误类型展示更准确的提示文案。

### 状态

已处理：

- 新增基金请求前端超时时间已放宽到 60 秒。
- 新增基金请求超时后会自动刷新基金列表，并提示用户后台可能仍已完成写入。

## 2. 删除基金时需要二次确认

### 现象

当前基金列表中点击“删除”会直接删除自选基金，没有弹出确认框。

### 影响

- 用户可能误点删除按钮。
- 删除操作缺少明确的风险确认。

### 后续处理建议

- 点击删除按钮时弹出确认框。
- 确认框展示基金代码和基金名称。
- 用户确认后再调用删除接口。
- 用户取消时不执行任何操作。

### 状态

已处理：

- 删除前会弹出确认框。
- 确认内容包含基金名称和基金代码。
- 确认框已改为页面内弹窗，避免使用浏览器原生确认框。

## 3. 联接基金和 QDII 基金持仓数据可能不完整

### 现象

基金 `012805` 使用 `akshare.fund_portfolio_hold_em()` 同步持仓时，当前同步到的 2025 年数据只有 4 条，并且每个报告期只有 1 条 `阅文集团`。

### 当前原因判断

- akshare 原始接口返回与数据库结果一致，不是当前解析或入库逻辑造成的异常。
- `012805` 是联接基金/QDII 基金，真实主要持仓可能体现在目标 ETF、基金资产或海外资产上。
- 当前使用的 `fund_portfolio_hold_em()` 主要获取股票持仓，可能无法完整反映联接基金/QDII 基金的底层持仓。

### 影响

- 持仓展示不完整。
- 后续估算净值时，如果仅依赖这类股票持仓，覆盖比例会很低，估算结果不可靠。

### 后续处理建议

- 针对联接基金、QDII 基金补充更合适的持仓数据源。
- 识别目标 ETF 或目标基金，并进一步解析其底层持仓。
- 在页面上展示持仓覆盖率或数据完整性提示。
- 对持仓比例为 0 或覆盖率过低的基金标记风险。

### 状态

已处理：

- 后端已按普通持仓源、ETF/QDII 目标基金持仓源做 fallback。
- 详情页会对联接基金、QDII 或持仓覆盖比例偏低的情况展示风险提示。

## 4. 持仓列表增加报告期筛选

### 现象

基金详情页当前展示该基金所有已同步的持仓记录，不能按报告期筛选。

### 需求

在“刷新持仓”按钮旁边增加一个报告期筛选控件。

### 约束

- 筛选项只能包含当前基金已有持仓数据的报告期。
- 如果当前基金没有持仓数据，则筛选控件置灰或隐藏。
- 默认可展示全部，或默认展示最新报告期。

### 后续处理建议

- 后端可继续返回全部持仓，由前端根据已有数据提取报告期列表并筛选。
- 或后端支持 `GET /api/funds/{fund_code}/holdings?report_period=2026Q1`。
- 报告期选项按时间倒序展示。

### 状态

已处理：

- 详情页已增加报告期筛选控件。
- 筛选项来自当前基金已有持仓数据，并按报告期倒序展示。

## 5. 接入并稳定化更多免费基金持仓数据源

### 现象

当前基金持仓主要依赖 `akshare`、东方财富/天天基金和易天富 ETF88。对部分联接基金、QDII 基金或 ETF 底层持仓，单一来源可能返回不完整、为空或页面结构变化。

### 需求

继续接入并验证以下免费数据源：

- 新浪基金。
- 基金速查网。
- 理杏仁。
- Investing。

### 影响

- 提高基金持仓和目标 ETF 映射的覆盖率。
- 降低单一数据源不可用时导致估算失败的概率。
- 为联接基金、QDII 基金、ETF 自身持仓提供更多校验来源。

### 后续处理建议

- 将每个站点封装为独立 `data_sources` adapter。
- 为每个来源记录 source 名称，便于前端和日志排查数据出处。
- 对非正式 API 或公开页面解析增加失败降级，避免单个来源异常影响整次刷新。
- 建立样例基金校验集，例如 `012805`、`008282`、`513380` 等。
- 对访问限制较多或页面结构不稳定的来源，先作为人工校验或低优先级 fallback。

### 状态

待处理。
